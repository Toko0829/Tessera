# Transcode

## Purpose

Turns an uploaded original into an adaptive-bitrate HLS ladder so any device on any
connection can stream it. Covers the Redis job queue (`libs/queue`) and the worker
(`apps/worker`) that consumes it and runs FFmpeg.

**Not responsible for:** accepting uploads (video-upload), serving playback (a later
module), or storage mechanics (the storage module).

## Public API

No HTTP endpoints; the worker never serves HTTP (CLAUDE.md section 5).

| Name | Kind | Responsibility |
| --- | --- | --- |
| `TranscodeQueue` | `libs/queue` | enqueue (API side), claim, complete, requeue in-flight |
| `TranscodeService.ProcessAsync(videoId)` | worker | one job end to end |
| `FfmpegTranscoder` | worker | ffprobe/ffmpeg as bounded external processes |
| `Worker` | hosted service | recover, poll, process one job at a time |

Output layout in storage: `videos/{videoId}/hls/master.m3u8` plus `v0_*`/`v1_*`
playlists and segments (720p 2500k and 480p 1200k, 4-second segments).

## Data Model

No new tables. Drives `Videos.Status`: `Uploaded` (queued) to `Processing` to
`Ready` or `Failed`, and records the ffprobe-measured `Videos.DurationSeconds`
together with `Ready` (a Ready video always has a length; the playback module
validates watch-progress positions against it). Queue state lives in two Redis
lists (`queue:transcode` and `queue:transcode:processing`).

## Dependencies

**Depends on:** `libs/persistence` (status transitions), `libs/storage` (download
original, upload renditions), `libs/queue`, Redis, and the `ffmpeg`/`ffprobe`
binaries (paths configurable; installed via the OS package manager locally, apt in
CI).
**Depended on by:** playback will read the HLS output; the API enqueues via
`libs/queue`.

## Security

- **Hostile input:** every ffprobe/ffmpeg run has a hard wall-clock ceiling and is
  killed past it; a non-zero exit fails the job. True OS-level sandboxing (container
  isolation, seccomp) is a deployment concern, documented here rather than claimed.
- **Metadata stripped:** `-map_metadata -1` removes GPS and device identifiers from
  the renditions (the original keeps its metadata and is never served).
- **No secrets in source:** storage and connection settings come from user-secrets
  or the secret store, like every other component.
- AuthN/authZ and rate limits do not apply: the worker has no request surface.

## Failure Modes

| What fails | Detection | Recovery | What the user sees |
| --- | --- | --- | --- |
| Undecodable input | ffmpeg/ffprobe non-zero exit | video marked `Failed`, original kept for diagnosis | Failed badge in the library |
| Runaway transcode | wall-clock timeout | process tree killed, video `Failed` | Failed badge |
| Worker crash mid-job | job left in the processing list | re-queued on next start; a `Processing` claim guard skips double work | stays `Processing` until restart |
| Graceful shutdown mid-job | cancellation | claim released back to `Uploaded`, job re-queued on start | briefly `Queued` again |
| Redis down | claim throws | logged, retried after the poll delay | uploads queue up |
| Enqueue fails after upload | complete request errors | row is already `Uploaded`; re-queueable | upload succeeded, stays `Queued` |

Known gap: a hard crash mid-job leaves that video `Processing` forever (the re-queued
job is skipped by the claim guard). A janitor that re-queues stale `Processing` rows
is future work, recorded here deliberately.

## Testing

Real end to end, no mocks: a two-second clip generated by ffmpeg itself is uploaded
to MinIO, processed by the real `TranscodeService`, and asserted `Ready` with the
master and both variant playlists present in storage. An undecodable input is
asserted `Failed`. The queue is exercised against real Redis, including in-flight
recovery. CI installs ffmpeg for these tests.

**Deliberately not covered:** the timeout kill path (would need a long-running input
and slow the suite); the bounded-process code is small and shared with the covered
failure path.

## Decisions

### Redis lists over Redis Streams or SQS

**Chose:** a two-list claim pattern (queue plus processing) on the Redis we already
run.
**Over:** Redis Streams consumer groups, or SQS now.
**Because:** one worker instance is the current reality; the two-list pattern gives
crash recovery with a fraction of the machinery, and the charter allows Redis-backed
queues when self-hosting. The queue is one small class behind which SQS or Streams
can land when scaling out demands it.
**Trade-off accepted:** single-worker recovery semantics, documented above.

### Two fixed renditions to start

**Chose:** 720p and 480p, H.264 plus AAC, 4-second segments.
**Over:** a source-adaptive ladder with more rungs.
**Because:** two rungs prove the whole pipeline and cover most connections; ladder
tuning is an optimisation with real data, not a guess to make now.
**Trade-off accepted:** small sources are upscaled to the fixed rungs; wasteful but
harmless, revisited with the ladder work.
